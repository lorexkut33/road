# Git
**git** - это инструмент для управления версиями, которая отслеживает изменения в файлах проекта и совместной работы над кодом 
Основные команды: 
- **git init** - создание локального репозитория(гит будет следить за файлами и писать, где были внесены изменения)
- **git add .** - добавление ВСЕХ файлов в "корзину"(можно добавлять по одному файлу просто после add [имя_файла])
- **git commit** - так называемый снимок кода
- **git push** - команда, которая отправляет наши коммиты в репозиторий
- **git pull** - забрать изменения с репозитория
- **git clone** - скачать файл из репозитория на компьютер
## Также главным будет то, что надо привязать гит к своему профилю
Чтобы привязать гит к своему профилю есть такие команды:
- **git config --global user.name "Имя"**
- **git config --global user.email "email@example.com"**
Достаточно один раз ввести, чтобы гит запомнил пользователя

## Чтобы привязать гит к удалённому репозиторию нужно ввести команду
Команду потребует при первом пуше в репозиторий:
- **git remote add origin https://github.com/username/repo.git**
Где вместо **username**, надо ввести логин твоего профиля

## Как запушить проект в репозиторий?
вот инструкция:
 1. Создаешь на гитхаб репозиторий
 2. Инициализируешь репозиторий: git init
 3. Добавь файлы: git add . (добавить все файлы).
 4. Зафиксируй изменения: git commit -m "Первый коммит".
 5. Привяжи к удаленному репозиторию: git remote add origin <ссылка_на_ваш_репозиторий>.
 6. Отправьте файлы: git push -u origin main (или master).

## Что означают буквы напротив файла проекта при создании локального репозитория?(как минимум в VS code)
Когда ты создаёшь новый файл, VS Code вешает на него метку U (Untracked) — Git его видит, но не отслеживает, ты ещё ни разу не делал git add для этого файла. После того как сделаешь git add, метка меняется на A (Added) — файл попал в staging и готов к коммиту. После git commit метка пропадает совсем — файл чистый, Git всё знает о нём.
Если потом изменишь уже отслеживаемый файл, появится M (Modified). Он бывает двух оттенков: оранжевый — изменения есть, но git add ещё не делал, зелёный — уже в staging. После коммита снова пропадает.
Остальные метки встречаются реже. D (Deleted) — файл удалён локально, но коммита ещё не было. R (Renamed) — файл переименован. C (Conflict) — конфликт при merge или rebase, нужно разрешать руками. I (Ignored) — файл попадает под правило в .gitignore и Git его не трогает.

## Ветка - что это такое?
**Ветка** - это одна из главных фич гита, которая позволяет вносить изменения для тестирования изменений в коде. Например, есть команда разработчиков и проект, над которым они работают. И чтобы не мешать друг другу, создают ветки для реализации своих задач, не затрагивая основную версию проекта. После всех изменений разработчик сливает ветку к главной(самособой после теста и если все хорошо)

Ветку создают командой:
- **git branch** [имя_ветки]
И чтобы перейти в только что созданную ветку, вводят такую коману:
- **git checkout** [имя_ветки]
Создать ветку и сразу переключиться на неё можно с помощью 1 команды, вместо 2:
- **git checkout -b** [имя_ветки] - ключ -b первая буква слова branch

## Как слить ветку с главной(обычно main/master)?
Слияние происходит после теста изменений, когда все изменения функционирую нормально, только тогда сливают в главную ветку
команда такая:
- **git checkout main** - надо перейти в главную ветку перед слиянием
- **git merge** [имя_ветки] - вместо [имя_ветки] пишем ветку, которую хотим слить
Также есть 2 способ слить ветки с помощью:
- **git rebase** [имя_ветки] - это команда по слиянию веток, но с переносом всех коммитов в одну ветку
Такой способ слияния через rebase удобен тем, что история коммитов будет последовательной и история будет чище

## Конфликт версий что это?
При слиянии веток возможен конфликт версий, например, Вася и Петя работают над одной задачей, не распределив задачи заранее(пример один из тысячи). Вася доделал свою задачу и решил слить ветки, но при слиянии возник конфликт версий. Потому что Петя уже загрузил свою задачу. И вот гит вам пишет что-то такое:
```git
 <<<<<<< HEAD
код Васи
=======
код Пети
>>>>>>> branch
```
**Что делать:**
 Оставить нужный вариант (или объединить)
 Удалить эти <<< === >>> строки
 Выполнить:
**git add .**
**git commit**

# Команда - противоположная ей команда
Здесь распианы все основные команды и противо положные им основные команды

**git init** → **git clone**
**git init** — создаёт новый репозиторий в текущей папке.
bashgit init
**git clone** — скачивает уже существующий репозиторий (противоположность init).
bashgit clone https://github.com/user/repo.git

|> 💡 clone автоматически делает init + скачивает все файлы + настраивает remote.


**git add** → **git restore / git reset**
**git add** — добавляет файл в staging area (подготовка к коммиту).
bashgit add file.txt       # конкретный файл
**git add .**              # всё в папке
**git restore** --staged — убирает файл из staging, но не трогает сами изменения.
bashgit restore --staged file.txt
**git reset** HEAD file.txt — то же самое, старый способ (до Git 2.23).

**Частая ошибка**: сделал git add . и случайно добавил лишний файл (например, .env с паролями).
Фикс: git restore --staged .env


**git commit** → git revert / git reset
**git commit** — сохраняет snapshot изменений в историю.
bashgit commit -m "добавил авторизацию"
**git revert** <hash> — создаёт новый коммит, который отменяет старый. Безопасно — история не ломается.
bashgit revert a1b2c3d
**git reset --hard** <hash> — переносит HEAD на нужный коммит и удаляет все коммиты после него. Опасно, если уже пушил.
bashgit reset --hard a1b2c3d

**Частая ошибка**: написал git commit -m "" или с опечаткой в сообщении.
Фикс: git commit --amend -m "правильное сообщение" — изменяет последний коммит.


**git push** → git pull / git fetch
**git push** — отправляет коммиты на удалённый репозиторий.
bashgit push origin main
**git pull** — скачивает изменения с remote и сразу сливает (merge) в текущую ветку.
bashgit pull origin main
**git fetch** — только скачивает изменения, но не сливает. Можно сначала посмотреть что пришло.
bashgit fetch origin
**git diff** main origin/main   # посмотреть разницу перед merge

**Частая ошибка**: git push выдаёт rejected — non-fast-forward.
Это значит на remote есть коммиты, которых нет у тебя локально.
Фикс: сначала git pull --rebase origin main, потом снова git push.


**Частая ошибка**: запушил случайно не в ту ветку.
Фикс: git push origin --delete wrong-branch удалит ветку на remote.


**git branch** → **git branch -d**
**git branch feature** — создаёт новую ветку.
bashgit branch feature-login
**git checkout** -b feature-login   # создать + сразу переключиться (короче)
**git switch** -c feature-login     # современный аналог
**git branch -d** feature-login — удаляет ветку (только если она уже слита).
bashgit branch -d feature-login     # безопасное удаление
**git branch -D** feature-login     # принудительное удаление

**Частая ошибка**: error: branch not fully merged.
Git предупреждает, что ветку не мёржили — данные могут потеряться.
Фикс: если точно не нужна — git branch -D. Если нужна — сначала смёрджи.


**git merge** → **git rebase** (альтернатива, не точная противоположность)
**git merge** — сливает ветку в текущую, создавая merge-коммит.
bashgit checkout main
**git merge** feature-login
**git rebase** — переносит коммиты поверх другой ветки. История линейная, но переписывается.
bashgit rebase main   # находясь в feature-ветке

**Частая ошибка**: конфликт при merge/rebase.
Гит покажет конфликтующие файлы

|> Руками редактируешь — оставляешь что нужно, убираешь маркеры <<<<, ====, >>>>.
|> Потом: git add file.txt → git commit (при merge) или git rebase --continue (при rebase).


**git stash** → **git stash pop**
**git stash** — прячет незакоммиченные изменения «в ящик», чтобы переключиться на другую ветку.
bashgit stash
**git stash push -m** "недоделанная авторизация"   # с названием
**git stash pop** — достаёт последние изменения обратно.
bashgit stash pop
**git stash list**              # посмотреть все stash-ы
**git stash apply stash@**{2}   # применить конкретный

**Частая ошибка**: сделал git stash pop и получил конфликт — разрешается так же, как при merge.


*Бонус — команды для диагностики*
Команда Что делает? **git status** показывает состояние файлов, **git log** --oneline краткая история коммитов, **git diff** что изменилось, но ещё не в staging, **git diff** --staged, что в staging, но ещё не закоммичено, **git remote** -v показывает remote-адреса
